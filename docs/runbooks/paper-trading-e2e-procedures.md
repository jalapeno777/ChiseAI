# Paper Trading E2E Validation Procedures

## Overview

This runbook defines end-to-end (E2E) validation procedures for the paper trading system. The procedures ensure that all components of the trading pipeline work correctly together, from signal generation through order execution, risk validation, notification delivery, and data persistence.

### Purpose

The E2E validation verifies:
- **Signal Flow Integrity**: Signals generated by strategies reach the execution engine
- **Order Lifecycle**: Orders progress correctly through Open → Fill → Close states
- **Risk Gate Enforcement**: Risk constraints are applied before order execution
- **LLM Provider Resilience**: Fallback mechanisms function when primary providers fail
- **Notification Delivery**: Discord messages are delivered for trading events
- **Data Persistence**: Trade data is correctly recorded in Redis, InfluxDB, and journal systems

### Scope

This runbook covers:
- 30-minute proof loop validation procedures
- Validation gates G1-G8 verification
- Evidence collection at 5-minute intervals (T0, T5, T10, T15, T20, T25, T30)
- Discord verification procedures
- Redis data validation commands
- InfluxDB query verification
- Troubleshooting common failures

### Prerequisites for Running E2E Validation

Before executing this runbook, ensure:
1. All paper trading services are operational (Redis, API, InfluxDB)
2. Discord bot is configured and connected
3. At least one strategy is active and generating signals
4. Paper trading mode is explicitly enabled
5. Risk gates are configured and operational

## E2E Test Scenarios

### Scenario 1: Signal Generation to Order Execution

This scenario validates the complete signal-to-order flow.

#### Test Steps

1. **Baseline Signal Count**
   ```bash
   # Record initial signal count
   T0_SIGNALS=$(redis-cli -p 6380 ZCARD paper:index:signals)
   echo "T0 Signal Count: $T0_SIGNALS"
   ```

2. **Wait for Signal Generation**
   ```bash
   # Wait for signal generation cycle (typically 1-5 minutes)
   sleep 180
   ```

3. **Verify Signal Creation**
   ```bash
   # Check new signal count
   T1_SIGNALS=$(redis-cli -p 6380 ZCARD paper:index:signals)
   SIGNAL_DELTA=$((T1_SIGNALS - T0_SIGNALS))
   echo "Signal Delta: $SIGNAL_DELTA"
   
   # PASS if SIGNAL_DELTA > 0
   ```

4. **Verify Order Creation**
   ```bash
   # Check if orders were created from signals
   T1_ORDERS=$(redis-cli -p 6380 ZCARD paper:index:orders)
   echo "Total Orders: $T1_ORDERS"
   
   # View recent orders
   redis-cli -p 6380 ZREVRANGE paper:index:orders 0 4 WITHSCORES
   ```

5. **Validation Criteria**
   - Signal delta > 0 (new signals generated)
   - Orders exist in paper:index:orders
   - Order timestamps correlate with signal timestamps

### Scenario 2: Order Lifecycle (Open → Fill → Close)

This scenario validates the complete order state progression.

#### Test Steps

1. **Create Test Order**
   ```bash
   # Submit a test paper order
   curl -X POST http://localhost:8001/api/v1/paper/orders \
     -H "Content-Type: application/json" \
     -d '{
       "symbol": "BTC-USDT",
       "side": "buy",
       "type": "market",
       "quantity": 0.001,
       "strategy_id": "e2e_test"
     }' | jq '.'
   ```

2. **Monitor Order Status**
   ```bash
   # Poll for order status (repeat every 10 seconds)
   ORDER_ID="<order_id_from_step_1>"
   
   curl -s "http://localhost:8001/api/v1/paper/orders/$ORDER_ID" | jq '{
     id: .id,
     status: .status,
     filled_qty: .filled_quantity,
     avg_price: .average_fill_price
   }'
   ```

3. **Verify Fill Creation**
   ```bash
   # Check for fills
   redis-cli -p 6380 ZREVRANGE paper:index:fills 0 4 WITHSCORES
   
   # Get fill details for specific order
   redis-cli -p 6380 KEYS "paper:fill:*" | grep "$ORDER_ID"
   ```

4. **Verify Outcome Recording**
   ```bash
   # Check outcome index
   redis-cli -p 6380 ZCARD paper:index:outcomes
   
   # View recent outcomes
   redis-cli -p 6380 ZREVRANGE paper:index:outcomes 0 4 WITHSCORES
   ```

5. **Lifecycle State Transitions**
   | State | Description | Validation Command |
   |-------|-------------|-------------------|
   | CREATED | Order submitted | `redis-cli -p 6380 HGET paper:order:<id> status` |
   | OPEN | Order accepted | Order exists in paper:index:orders |
   | PARTIAL_FILL | Partial execution | `redis-cli -p 6380 HGET paper:fill:<id> filled_quantity` |
   | FILLED | Fully executed | Fill exists in paper:index:fills |
   | CLOSED | Outcome recorded | Outcome exists in paper:index:outcomes |

### Scenario 3: Risk Gate Validation

This scenario validates that risk constraints are enforced.

#### Test Steps

1. **Check Risk Configuration**
   ```bash
   # Verify risk gates are enabled
   curl -s http://localhost:8001/api/v1/risk/config | jq '{
     max_position_size: .max_position_size,
     max_exposure_pct: .max_exposure_pct,
     daily_loss_limit: .daily_loss_limit
   }'
   ```

2. **Attempt Exceeding Risk Limits**
   ```bash
   # Try to create an order exceeding position limits
   curl -X POST http://localhost:8001/api/v1/paper/orders \
     -H "Content-Type: application/json" \
     -d '{
       "symbol": "BTC-USDT",
       "side": "buy",
       "type": "market",
       "quantity": 100.0,
       "strategy_id": "e2e_risk_test"
     }' | jq '{
       success: .success,
       error: .error,
       rejection_reason: .rejection_reason
     }'
   
   # Expected: rejection with risk constraint violation
   ```

3. **Verify Risk Event Logging**
   ```bash
   # Check for risk events in Redis
   redis-cli -p 6380 KEYS "paper:risk_event:*" | tail -5
   
   # Get risk event details
   redis-cli -p 6380 HGETALL "paper:risk_event:<latest>"
   ```

4. **Validation Criteria**
   - Order exceeding limits is rejected
   - Rejection reason specifies risk constraint
   - Risk event is logged to Redis
   - No position created for rejected order

## Validation Gates (G1-G8)

### Gate Overview

The validation gates provide systematic verification of the paper trading system. Each gate validates specific components and data flows.

| Gate | Name | Description | Tracks Validated |
|------|------|-------------|------------------|
| G1 | Signal Generation | Signals are being generated | E1 |
| G2 | Order Creation | Orders are being created from signals | S1, E2 |
| G3 | Fill Execution | Fills are occurring | S3, E4 |
| G4 | Outcome Capture | Outcomes are being recorded | S4 |
| G5 | Discord Delivery | Discord notifications are sent | S5 |
| G6 | InfluxDB Orders | Order data is in InfluxDB | S2, S3 |
| G7 | Canary Status | Canary deployment is healthy | S6 |
| G8 | Burn-in Verdict | Overall burn-in test verdict | S1-S6 |

### G1: Signal Generation

**Purpose**: Verify that signals are being generated and indexed.

**Validation Logic**:
```bash
# Get signal count
SIGNAL_COUNT=$(redis-cli -p 6380 ZCARD paper:index:signals)
echo "Total Signals: $SIGNAL_COUNT"

# PASS if SIGNAL_COUNT > 0
[ "$SIGNAL_COUNT" -gt 0 ] && echo "G1: PASS" || echo "G1: FAIL"
```

**Evidence Required**:
- Signal count from Redis
- Sample signal details
- Timestamp of most recent signal

**Troubleshooting**:
- If FAIL: Check strategy configuration and scheduler status
- Verify `paper:index:signals` sorted set exists
- Check strategy heartbeat in Redis

### G2: Order Creation

**Purpose**: Verify that orders are being created from signals.

**Validation Logic**:
```bash
# Get order count
ORDER_COUNT=$(redis-cli -p 6380 ZCARD paper:index:orders)
echo "Total Orders: $ORDER_COUNT"

# Get recent orders
redis-cli -p 6380 ZREVRANGE paper:index:orders 0 4 WITHSCORES

# PASS if ORDER_COUNT > 0 and delta > 0 over window
[ "$ORDER_COUNT" -gt 0 ] && echo "G2: PASS" || echo "G2: FAIL"
```

**Evidence Required**:
- Order count from Redis
- Sample order details (symbol, side, quantity)
- Order-to-signal correlation

**Troubleshooting**:
- If FAIL: Check execution engine logs
- Verify risk gates are not blocking all orders
- Check order creation API health

## 30-Minute Proof Loop Procedure

### Overview

The 30-minute proof loop validates the paper trading system through continuous operation with periodic evidence collection.

### Procedure Steps

#### Pre-Loop Setup (T-5 minutes)

1. **Verify Environment**
   ```bash
   # Run pre-loop checks
   ./scripts/validation/paper_pre_loop_check.sh
   
   # Verify all services healthy
   curl -s http://localhost:8001/api/v1/health | jq '.status'
   ```

2. **Record Baseline Metrics**
   ```bash
   # Capture T0 baseline
   T0_EVIDENCE="evidence/T0_$(date +%Y%m%d_%H%M%S).json"
   ./scripts/validation/paper_collect_evidence.sh --output="$T0_EVIDENCE"
   ```

3. **Verify Active Strategies**
   ```bash
   # Check strategy status
   curl -s http://localhost:8001/api/v1/strategies/active | jq '.strategies[] | {
     id: .id,
     name: .name,
     status: .status
   }'
   ```

#### T0: Loop Start

```bash
# Start the 30-minute timer
START_TIME=$(date +%s)
echo "Proof Loop Started: $(date -d @$START_TIME)"

# Record T0 evidence
./scripts/validation/paper_collect_evidence.sh --time=T0 --output=evidence/T0.json
```

#### T5: First Checkpoint (5 minutes)

```bash
# At T+5 minutes, collect evidence
./scripts/validation/paper_collect_evidence.sh --time=T5 --output=evidence/T5.json

# Quick gate check
cat evidence/T5.json | jq '.gates | {g1, g2}'
```

**Checkpoint Criteria**:
- G1 should show signal activity (PASS)
- G2 should show order creation (PASS or in-progress)

#### T10: Second Checkpoint (10 minutes)

```bash
# At T+10 minutes, collect evidence
./scripts/validation/paper_collect_evidence.sh --time=T10 --output=evidence/T10.json

# Verify G3 (fills) may be showing activity
cat evidence/T10.json | jq '.gates | {g2, g3}'
```

**Checkpoint Criteria**:
- G2 should be PASS
- G3 may be PASS if fills have occurred

#### T15: Midpoint Checkpoint (15 minutes)

```bash
# At T+15 minutes, collect evidence
./scripts/validation/paper_collect_evidence.sh --time=T15 --output=evidence/T15.json

# Run gate validation
./scripts/validation/paper_gates_check.sh --input=evidence/T15.json
```

**Midpoint Criteria**:
- G1, G2, G3 should be PASS
- G4 may be PASS if outcomes recorded

#### T20: Fourth Checkpoint (20 minutes)

```bash
# At T+20 minutes, collect evidence
./scripts/validation/paper_collect_evidence.sh --time=T20 --output=evidence/T20.json

# Check Discord activity
python3 scripts/validation/paper_discord_verify.py --time=T20
```

**Checkpoint Criteria**:
- G1-G4 should be PASS
- G5 (Discord) may show activity

#### T25: Pre-Completion Checkpoint (25 minutes)

```bash
# At T+25 minutes, collect evidence
./scripts/validation/paper_collect_evidence.sh --time=T25 --output=evidence/T25.json

# Full gate check
./scripts/validation/paper_gates_check.sh --input=evidence/T25.json
```

**Checkpoint Criteria**:
- All automated gates should be PASS
- G6 may be INFO (InfluxDB lag)

#### T30: Loop Completion

```bash
# At T+30 minutes, collect final evidence
./scripts/validation/paper_collect_evidence.sh --time=T30 --output=evidence/T30.json

# Calculate deltas
./scripts/validation/paper_calculate_deltas.sh --baseline=evidence/T0.json --final=evidence/T30.json

# Generate burn-in verdict
./scripts/validation/paper_burn_in_verdict.sh --evidence-dir=evidence/ --output=verdict.json
```

### Time Checkpoint Summary

| Time | Action | Gates to Check |

| T5 | First checkpoint | G1, G2 |
| T10 | Second checkpoint | G2, G3 |

## Troubleshooting Common Failures

### G1 Failure: No Signal Generation

**Symptoms**:
- `paper:index:signals` count is 0 or not increasing
- G1 validation returns FAIL

**Diagnosis**:
```bash
# Check strategy status
curl -s http://localhost:8001/api/v1/strategies/active | jq '.'

# Check scheduler heartbeat
redis-cli -p 6380 HGETALL bmad:chiseai:scheduler:heartbeat

# Check for errors in logs
docker logs chiseai-api --tail 100 | grep -i "signal\|error"
```

**Resolution**:
1. Verify at least one strategy is active
2. Check scheduler is running and has recent heartbeat
3. Review strategy configuration for errors
4. Restart signal generation if necessary

### G2 Failure: No Order Creation

**Symptoms**:
- Signals exist but no orders created
- G2 validation returns FAIL

**Diagnosis**:
```bash
# Check execution engine status
curl -s http://localhost:8001/api/v1/execution/status | jq '.'

# Check risk gate configuration
curl -s http://localhost:8001/api/v1/risk/config | jq '.'

# Review order creation logs
docker logs chiseai-api --tail 200 | grep -i "order\|reject"
```

**Resolution**:
1. Verify execution engine is in paper mode
2. Check risk gates are not blocking all orders
3. Verify signal-to-order pipeline connectivity
4. Check for order validation errors

### G3 Failure: No Fill Execution

**Symptoms**:
- Orders exist but no fills recorded
- G3 validation returns FAIL

**Diagnosis**:
```bash
# Check order status
redis-cli -p 6380 HGET paper:order:<order_id> status

# Check exchange connectivity
curl -s http://localhost:8001/api/v1/exchanges/status | jq '.'

# Review fill processing logs
docker logs chiseai-api --tail 200 | grep -i "fill\|execution"
```

**Resolution**:
1. Verify exchange connectivity
2. Check order prices are within market range
3. Review fill processing configuration
4. Verify fill callback handlers are registered

### G4 Failure: No Outcome Capture

**Symptoms**:
- Fills exist but no outcomes recorded
- G4 validation returns FAIL

**Diagnosis**:
```bash
# Check OutcomePersistence service
curl -s http://localhost:8001/api/v1/services/outcome-persistence/status | jq '.'

# Verify Redis connection for outcomes
redis-cli -p 6380 PING

# Review outcome persistence logs
docker logs chiseai-api --tail 200 | grep -i "outcome\|persistence"
```

**Resolution**:
1. Verify OutcomePersistence service is running
2. Check Redis connectivity from outcome service
3. Review outcome processing configuration
4. Verify sorted set `paper:index:outcomes` exists

### G5 Failure: Discord Not Delivered

**Symptoms**:
- Trading activity exists but no Discord messages
- G5 validation returns FAIL

**Diagnosis**:
```bash
# Check Discord bot status
python3 scripts/validation/paper_discord_verify.py --status

# Verify Discord token
echo $DISCORD_BOT_TOKEN | wc -c

# Review Discord bot logs
docker logs chiseai-api --tail 200 | grep -i "discord\|notification"
```

**Resolution**:
1. Verify Discord bot token is valid and not expired
2. Check Discord channel IDs are correct
3. Verify bot has permissions to post in channels
4. Check Discord API rate limits

### G6 INFO: InfluxDB Query Issues

**Symptoms**:
- G6 returns INFO instead of PASS
- Redis shows data but InfluxDB queries return empty

**Diagnosis**:
```bash
# Check InfluxDB connectivity
curl -s http://localhost:18087/ping

# Verify database exists
curl -s "http://localhost:18087/query?q=SHOW+DATABASES"

# Check InfluxDB retention policy
curl -s "http://localhost:18087/query?db=paper_trading&q=SHOW+RETENTION+POLICIES"
```

**Note**: G6 INFO is acceptable when Redis data exists, as Redis is the canonical source.

### G7 Failure: Canary Status Missing

**Symptoms**:
- No canary data in InfluxDB
- G7 validation returns FAIL

**Diagnosis**:
```bash
# Check canary configuration
curl -s http://localhost:8001/api/v1/canary/config | jq '.'

# Verify canary metrics emission
curl -s "http://localhost:18087/query?db=paper_trading&q=SHOW+MEASUREMENTS" | grep canary
```

**Resolution**:
1. Verify canary deployment is enabled
2. Check canary metrics configuration
3. Verify InfluxDB measurement exists

## Sign-off Criteria

### Minimum Pass Requirements

For a successful 30-minute proof loop, the following criteria must be met:

| Criterion | Requirement | Evidence |
|-----------|-------------|----------|
| Duration | Full 30 minutes completed | T0 and T30 timestamps |
| G1 (Signals) | PASS | Signal count > 0 |
| G2 (Orders) | PASS | Order count > 0 |
| G3 (Fills) | PASS | Fill count > 0 |
| G4 (Outcomes) | PASS | Outcome count > 0 |
| G5 (Discord) | PASS or MANUAL | Message IDs or manual verification |
| G6 (InfluxDB) | PASS or INFO | Query results or Redis validation |
| G7 (Canary) | PASS | Canary data exists |
| G8 (Verdict) | PASS | Burn-in verdict document |

### Automated Gate Threshold

- **Minimum**: 6/8 automated gates PASS
- **G5 Exception**: MANUAL is acceptable per acceptance criteria
- **G6 Exception**: INFO is acceptable with documented justification

### Evidence Completeness

All required evidence must be present:
- [ ] T0 baseline evidence
- [ ] T5 checkpoint evidence
- [ ] T10 checkpoint evidence
- [ ] T15 checkpoint evidence
- [ ] T20 checkpoint evidence
- [ ] T25 checkpoint evidence
- [ ] T30 final evidence
- [ ] Delta calculations
- [ ] Burn-in verdict

### Sign-off Process

1. **Verify Evidence Completeness**
   ```bash
   # Check all evidence files exist
   ls -la evidence/
   
   # Verify evidence integrity
   ./scripts/validation/paper_verify_evidence.sh --dir=evidence/
   ```

2. **Review Gate Results**
   ```bash
   # Display gate summary
   cat verdict.json | jq '.gate_results'
   ```

3. **QA Sign-off**
   - QA engineer reviews evidence bundle
   - Verifies sign-off criteria met
   - Documents any exceptions or notes

4. **Approval Documentation**
   ```bash
   # Generate sign-off document
   ./scripts/validation/paper_generate_signoff.sh \
     --verdict=verdict.json \
     --qa-engineer="<name>" \
     --output=signoff.md
   ```
