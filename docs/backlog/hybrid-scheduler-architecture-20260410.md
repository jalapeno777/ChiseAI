# Hybrid Scheduler Architecture — Canary Scoreboard

> **Date**: 2026-04-10  
> **Status**: Implemented

## 1. Architecture Overview

```
┌─────────────────────────────────────────────────────────────┐
│                  CANARY SCOREBOARD DELIVERY                 │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  RUNTIME SCHEDULER (primary)                         │  │
│  │  - Runs on host/cron/systemd timer                  │  │
│  │  - Reads .env for DISCORD_WEBHOOK_URL               │  │
│  │  - Exec: python3 scripts/ops/canary_scoreboard_discord.py  │
│  │  - Audit: _bmad-output/ci/canary-scoreboard-discord.audit.json │
│  │  - Status: _bmad-output/ci/canary-scoreboard-discord.status   │
│  │  - Modes: live | --dry-run | --test                  │  │
│  └──────────────────────────────────────────────────────┘  │
│                           │                                │
│                           │ (heartbeat / watchdog)          │
│                           ▼                                │
│  ┌──────────────────────────────────────────────────────┐  │
│  │  WOODPECKER CRON FALLBACK (secondary)                 │  │
│  │  - .woodpecker/cron-eval.yaml canary-scoreboard-discord step  │
│  │  - Runs on 6h-eval and daily-trends crons            │  │
│  │  - Always exits 0; emits DRY-RUN if webhook missing  │  │
│  │  - Purpose: compliance visibility, not delivery      │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## 2. Runtime Scheduler Commands

### Normal run (host environment)

```bash
# From project root, reads .env for secrets
python3 scripts/ops/canary_scoreboard_discord.py
```

### Dry-run (safe test, no webhook call)

```bash
python3 scripts/ops/canary_scoreboard_discord.py --dry-run
```

### Test mode (uses DISCORD_TEST_WEBHOOK_URL)

```bash
python3 scripts/ops/canary_scoreboard_discord.py --test
```

## 3. Audit Artifacts

| Artifact    | Path                                                    | Purpose                                        |
| ----------- | ------------------------------------------------------- | ---------------------------------------------- |
| Status      | `_bmad-output/ci/canary-scoreboard-discord.status`      | PASS/DRY-RUN/ERROR                             |
| Audit log   | `_bmad-output/ci/canary-scoreboard-discord.audit.json`  | Timestamp, mode, webhook status, HTTP response |
| Test status | `_bmad-output/ci/canary-scoreboard-discord.test.status` | PASS/DRY-RUN/ERROR (test mode)                 |

## 4. Fallback Behavior (Woodpecker)

The `canary-scoreboard-discord` step in `.woodpecker/cron-eval.yaml`:

- Uses `--dry-run` flag to prevent CI failures
- Always exits 0 (by design)
- Emits DRY-RUN status if `DISCORD_WEBHOOK_URL` is not in CI env
- Provides compliance heartbeat only

## 5. Canary Data Source

The script reads from `data/canary/*.json` looking for active canaries with status field in `("active", "running", "canary")`.

Expected data shape:

```json
{
  "name": "R2a",
  "status": "active",
  "total_trades": 142,
  "net_profit": 1234.56,
  "win_rate": 0.62,
  "max_drawdown": 0.08,
  "description": "R2a Canary Period 2026-04-08 to 2026-04-29"
}
```

## 6. Security Considerations

- `.env` is gitignored — never commit webhook URLs
- Runtime scheduler has access to host secrets via .env
- Woodpecker CI does NOT have access to CI-level secrets for this webhook
- DRY-RUN mode ensures no accidental notifications from CI
- Test webhook (`DISCORD_TEST_WEBHOOK_URL`) used for validation

## 7. Future Sessions

When monitoring or debugging the canary scoreboard:

1. Check runtime audit artifact: `_bmad-output/ci/canary-scoreboard-discord.audit.json`
2. Check runtime status: `_bmad-output/ci/canary-scoreboard-discord.status`
3. If runtime scheduler is failing, fix and redeploy to host environment
4. Woodpecker fallback only provides visibility — it is NOT the authoritative path
